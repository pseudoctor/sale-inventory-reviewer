# 库存分析系统运维操作手册（最终版）

## 1. 目标与适用范围
本手册用于保障库存分析系统长期稳定运行，覆盖以下场景：
- 单系统或多系统批量出报表
- 日常数据更新与结果分发
- 运行失败排障与恢复
- 配置变更、回归验证与发布

适用角色：
- 业务数据管理员：准备销售/库存/装箱因子数据
- 运维执行人员：运行脚本、核查结果、处理异常
- 一线使用人员（销售/配送/理货）：消费报表并执行动作

## 2. 目录规范（必须遵守）
- `raw_data/`：零售系统原始数据目录（每个系统一个子目录）
- `data/`：基础映射数据（当前为 `sku装箱数.xlsx`）
- `reports/`：输出报表与批量汇总
- `config.yaml`：运行配置
- `run.sh` / `run.bat`：标准启动入口

多系统目录示例：
- `raw_data/陕西华润/`
- `raw_data/宁夏物美/`
- `raw_data/甘肃华润/`

子目录文件建议：
- 销售：`销售YYYYMM.xlsx`（或 `.xls`）
- 库存：`库存.xlsx`（或 `.xls`）

说明：文件名不强制带系统名前缀。系统归属由 `data_subdir` 决定。

## 3. 配置规则（核心）
`config.yaml` 关键字段：
- `run_mode`: `single` 或 `batch`
- `raw_data_dir`: 数据根目录
- `carton_factor_file`: 装箱因子文件，默认 `./data/sku装箱数.xlsx`
- `season_mode`:
  - `false`：淡季，取保守口径（低）
  - `true`：旺季，取积极口径（高）
- `batch.continue_on_error`:
  - `true`：某系统失败不影响其他系统
  - `false`：首个失败即停止
- `batch.systems[*].enabled`:
  - `true`：本次参与生成
  - `false`：跳过（summary 显示 `SKIPPED`）

## 4. 每日标准操作流程（SOP）
1. 更新数据
- 把各系统销售和库存文件放到对应 `raw_data/<系统名>/`
- 更新 `data/sku装箱数.xlsx`（如 SKU 有变化）

2. 快速预检
```bash
python3 scripts/health_check.py
```

3. 生成报表
- macOS/Linux：
```bash
./run.sh
```
- Windows：
```bat
run.bat
```

4. 核查批量结果
- 打开 `reports/batch_run_summary.xlsx`
- 检查每个系统状态是否为 `SUCCESS`

5. 分发报表
- 各系统报表默认命名：`<系统名><库存日期>库存预警.xlsx`
- 例如：`陕西华润20260209库存预警.xlsx`

## 5. 输出结果解释（业务必须掌握）
核心 sheet：
- `明细`：完整 SKU 级分析（风险、周转、补货、调出）
- `建议补货清单`：建议补货数量 > 0
- `建议调货清单`：建议调出数量 > 0
- `缺货清单`：有销量且库存为 0
- `库存缺失SKU清单`：销售有记录但库存表缺失该 SKU
- `运行状态`：数据窗口、忽略文件、缺失映射、程序版本

补货箱数规则：
- 列名：`建议补货箱数`
- 格式：`3件`
- 公式：`建议数量 / 装箱数（因子）`
- 旺季 `season_mode=true`：向上取整（ceil）
- 淡季 `season_mode=false`：向下取整（floor）

## 6. 故障处理手册（按优先级）
### 6.1 依赖缺失（最常见）
现象：`.xls` 读取失败，提示缺少 `xlrd`。  
处理：
```bash
python3 -m pip install -r requirements.lock
```
若 lock 不存在：
```bash
python3 -m pip install -r requirements.txt
```

### 6.2 单系统 FAILED
先看 `reports/batch_run_summary.xlsx`：
- `message`
- `error_stage`
- `loaded_sales_files`
- `missing_sales_files`
- `inventory_file_exists`

常见原因：
- 配置文件名与实际文件不一致
- 列名不符合识别规则
- 日期字段无法解析

### 6.3 自动扫描未匹配销售文件
处理顺序：
1. 优先在配置里显式写 `sales_files`
2. 文件名按 `销售YYYYMM` 规范
3. 在报表 `运行状态` 查看“自动扫描忽略销售文件”

## 7. 变更与发布流程
配置或代码变更后，按顺序执行：
```bash
python3 -m ruff check .
python3 -m pytest -q
python3 scripts/health_check.py
./run.sh
```

发布前检查：
- 关键系统报表成功生成
- `batch_run_summary.xlsx` 无 `FAILED`
- `运行状态` 无关键告警（窗口无重叠、日期解析失败等）

## 8. 多系统运维动作
新增系统：
1. 创建目录 `raw_data/<系统名>/`
2. 在 `batch.systems` 新增条目（含 `display_name`、`data_subdir`、文件名）
3. 设 `enabled: true`

停用系统（推荐）：
- 将 `enabled` 改为 `false`

永久移除系统：
- 删除 `batch.systems` 对应条目

## 9. 稳定性与可复现建议
- 销售文件尽量使用显式 `sales_files`，减少自动扫描歧义
- 每次运行后归档 `batch_run_summary.xlsx`
- 保留当日 `config.yaml` 快照（便于复盘）
- 不要手工改报表计算列，任何规则调整都在代码或配置中完成

## 10. 现场最小排障法（30 秒）
1. 看 `reports/batch_run_summary.xlsx` 是否有 `FAILED`
2. 如果失败，先看 `error_stage` 和 `message`
3. 再打开对应系统报表的 `运行状态` sheet
4. 按提示修数据或配置后重跑

---
单条原则：先看 `batch_run_summary.xlsx`，再看系统报表的 `运行状态`，绝大多数问题可在两步内定位。
